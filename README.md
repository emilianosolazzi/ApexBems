## Safety, Auditability & Shadow Mode

ApexBEMS now includes production-oriented safety and audit layers for shadow-mode validation.

| Layer | Purpose |
|---|---|
| `SafetyGateway` | Blocks unsafe dispatch before state update or market/control submission |
| `TelemetryAdapter` | Converts read-only site and miner telemetry into deterministic safety snapshots and hashes |
| `VirtualSiteSimulator` | Exercises dry-run PCS/miner command artifacts, acknowledgements, telemetry feedback, and audit persistence before live hardware integration |
| `AuditLogger` | Records dispatch context, safety fields, optimizer status, hashes, and replay metadata |
| `MetricsRegistry` | Tracks optimizer success/failure, fallback intervals, SOC safety counters, and blocked submissions |
| `SchemaValidation` | Validates inbound/outbound payload contracts and rejects ambiguous units |
| `shadow_mode=True` | Default mode for read-only recommendation workflows |
| `allow_market_submission=False` | Prevents accidental bid submission by default |

The controller blocks unsafe dispatch and safe-fallback dispatch before market submission. Market submission is disabled by default unless explicitly configured.

## Validation Boundary

The public benchmark proves optimizer behavior on public ERCOT and Bitcoin data. The virtual-site HIL simulator proves command contracts, safety gating, acknowledgements, telemetry feedback, and audit persistence in dry-run mode. These do **not** prove direct private-site control readiness.

Production deployment requires:

1. Read-only shadow-mode validation against private PCS, miner, breaker, meter, transformer, feeder, tariff, and SCADA telemetry.
2. Live hardware-in-the-loop testing against real site equipment.
3. Cryptographically signed dry-run commands and replayable site snapshots.
4. Operator approval workflows.
5. Staged activation with emergency stop, stale-data lockout, and manual override.

Mining-load co-optimization is implemented in the optimizer/controller path for shadow-mode analysis. Live miner-manager control remains outside the public benchmark and requires private telemetry, signed dry-run commands, live hardware-in-the-loop testing, and staged activation.

## Current Limitations

- Mining co-optimization is implemented for shadow-mode storage + mining dispatch, but live miner-manager control is not included.
- The public benchmark uses public ERCOT/BTC data, not private site telemetry.
- Read-only mapping/replay telemetry adapters are implemented; live vendor drivers for PCS, miners, meters, breakers, transformers, feeders, and SCADA are not included.
- Hardware control is intentionally disabled by default.
- Market submission is disabled by default.
- Production control requires shadow-mode site validation, cryptographically signed dry-run commands, live hardware-in-the-loop testing, and staged activation.

## Scientific Foundations

ApexBEMS implements state-of-the-art methods from:

- **Stochastic Model Predictive Control** — scenario-based dispatch optimization over uncertain price paths
- **Linear Programming** — dispatch optimization with explicit operational constraints
- **Bayesian Model Averaging** — ensemble forecast weighting
- **SHAP Explainability** — optional feature-importance reporting when SHAP is available
- **Arrhenius-Inspired Degradation Models** — temperature-aware battery aging cost estimates

Note: battery temperature is currently used in degradation-cost calculations. The modular safety layer can block recommendations on transformer thermal limits when telemetry is provided; full battery thermal derating remains a future model extension.
