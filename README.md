## Key Features

### Core Capabilities

| Feature | Description |
|---|---|
| **Stochastic MPC** | Monte Carlo optimization with volatility-aware horizon adjustment |
| **Flexible-Load Integration Path** | Storage optimizer with schema contracts and benchmark support for future ASIC/GPU load integration |

|---|---|
| **Ensemble Forecaster** | Bayesian softmax regret weighting across LSTM (attention), Prophet, and XGBoost |
| **Drift Detection** | Automatic retraining triggered by forecast error thresholds |
| **Uncertainty Quantification** | VAE-based scenario generation for risk-adjusted dispatch |

### Auditing & Compliance

| Feature | Description |
|---|---|

| **Bottleneck Analysis** | Identification of limiting constraints such as SOC and ramping |

## System Architecture

```text
┌──────────────────────────────────────────────────────────┐
│                     Event Bus / Services                 │
├────────────┬────────────┬────────────┬──────────┬────────┤
│ Forecaster │ Optimizer  │ Battery    │ Monitor  │ Market │
│ Service    │ Service    │ State      │ Service  │ Client │
├────────────┴────────────┴────────────┴──────────┴────────┤
│                    MPC Controller                         │
├──────────────────────────────────────────────────────────┤
│ SafetyGateway │ PolicyValidator │ SchemaValidation        │
├──────────────────────────────────────────────────────────┤
│ AuditLogger   │ MetricsRegistry │ Replayable Benchmarking │
└──────────────────────────────────────────────────────────┘
```

| Component | Responsibility |
|---|---|
| `ForecasterService` | Multi-model ensemble prediction, VAE scenario generation |
| `OptimizationService` | Multi-market stochastic LP, parametric bid curves, scenario reduction |
| `BatteryStateService` | SOC tracking, state persistence, and degradation cost support |
| `MonitoringService` | Drift detection, dynamic horizon, bandit exploration |
| `MarketAPIClient` | Price data retrieval and optional bid submission; submission is disabled by default |
| `SafetyGateway` | Validates finite dispatch, power limits, raw SOC, clipped SOC, and SOC violations before action |
| `PolicyValidator` | Pre-submission bid feasibility checks |
| `SchemaValidation` | Validates telemetry, market feed, dispatch, PCS, and mining-control contracts |
| `AuditLogger` | Structured dispatch logging with explainability metadata |
| `MetricsRegistry` | Tracks optimizer, fallback, SOC safety, benchmark, and submission counters |
| `EventBus` | Async-aware pub/sub for decoupled inter-component communication |

## Setup & Run

Python 3.12 is the recommended runtime for the full forecasting stack, including TensorFlow. Python 3.13 can run the core environment, but TensorFlow is intentionally skipped by `requirements.txt` on Python 3.13 and newer.

On Windows PowerShell:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
```

Recommended public benchmark run:

```powershell
.\.venv\Scripts\python.exe scripts\run_public_benchmark.py --node hbHubAvg --market rt --horizon-steps 8 --miner-power-kw 1000 --miner-efficiency-j-per-th 25
```

To run the preserved legacy monolith demo:

```powershell
python ApexBEMS.py
```

To verify the environment without running the full demo:

```powershell
python -m py_compile apex_bems/config.py apex_bems/battery.py apex_bems/optimization.py
python -m compileall apex_bems
python -m pip check
python -m pytest -v
```

Legacy monolith check:

```powershell
python -m py_compile ApexBEMS.py
python -c "import ApexBEMS; print('ApexBEMS import ok')"
```

The repository keeps `ApexBEMS.py` as the preserved legacy monolith and also provides the modular `apex_bems` package for new development. The schema files under `schemas/` are JSON Schema documents stored with `.py` extensions; treat them as contract documents, not Python modules.

## Safety, Auditability & Shadow Mode

ApexBEMS now includes production-oriented safety and audit layers for shadow-mode validation.

| Layer | Purpose |
|---|---|
| `SafetyGateway` | Blocks unsafe dispatch before state update or market/control submission |
| `AuditLogger` | Records dispatch context, safety fields, optimizer status, hashes, and replay metadata |
| `MetricsRegistry` | Tracks optimizer success/failure, fallback intervals, SOC safety counters, and blocked submissions |
| `SchemaValidation` | Validates inbound/outbound payload contracts and rejects ambiguous units |
| `shadow_mode=True` | Default mode for read-only recommendation workflows |
| `allow_market_submission=False` | Prevents accidental bid submission by default |

The controller blocks unsafe dispatch and safe-fallback dispatch before market submission. Market submission is disabled by default unless explicitly configured.

## Configuration & Monitoring

All behavior is controlled through `BEMSConfig`:

| Parameter | Default | Description |
|---|---|---|
| `capacity_kwh` | `1000.0` | Asset storage capacity |
| `max_power_kw` | `250.0` | Max charge/discharge power |
| `horizon_hours` | `4` | MPC look-ahead window |
| `max_scenarios` | `100` | Monte Carlo scenario count |
| `solver` | `"CBC"` | Optimization backend (CBC, Gurobi, CPLEX, HiGHS) |
| `confidence` | `0.95` | Risk-adjusted optimization parameter |
| `dt_hr` | `5 / 60` | Dispatch interval in hours |
| `retrain_freq_steps` | `288` | Scheduled model retraining cadence |
| `state_file` | `"battery_state.json"` | Battery state persistence file |
| `audit_db` | `"audit_log.db"` | SQLite audit ledger path |
| `physical_dispatch_link` | `True` | Forces optimizer physical dispatch to match SOC-driving power |
| `enforce_price_unit_check` | `True` | Rejects likely `$/MWh` inputs when optimizer expects `$/kWh` |
| `shadow_mode` | `True` | Keeps controller in recommendation-only mode by default |
| `allow_market_submission` | `False` | Prevents accidental market submission unless explicitly enabled |

### Quick Start

```python
import asyncio
from datetime import datetime

import pandas as pd

from apex_bems import BEMSConfig, main_services

async def run():
    cfg = BEMSConfig(
        capacity_kwh=1000.0,
        max_power_kw=250.0,
        market="ERCOT",
        solver="CBC",
    )
    mpc = main_services(cfg)   # validates config, wires all services

    for step in range(100):
        current_time = datetime.now()
        market_data = pd.DataFrame(
            {"price": [0.045]},
            index=[current_time],
        )
        plan = await mpc.step(current_time, market_data)
        if plan is not None:
            print(f"Dispatch: {plan.action_kw:+.1f} kW | SOC: {plan.soc:.1%} | {plan.reason}")

asyncio.run(run())
```

## Validation Boundary

The public benchmark proves optimizer behavior on public ERCOT and Bitcoin data. It does **not** prove direct private-site control readiness.

Production deployment requires:

1. Read-only shadow-mode validation against private PCS, miner, breaker, meter, tariff, and SCADA telemetry.
2. Hardware-in-the-loop testing.
3. Signed dry-run commands and replayable site snapshots.
4. Operator approval workflows.
5. Staged activation with emergency stop, stale-data lockout, and manual override.

Current mining support is a **break-even proxy**, not LP co-optimized mining-load control.

---

## Current Limitations

- Mining is currently modeled as a break-even proxy, not fully co-optimized inside the storage LP.
- The public benchmark uses public ERCOT/BTC data, not private site telemetry.
- Hardware control is intentionally disabled by default.
- Market submission is disabled by default.
- Production control requires shadow-mode site validation, signed dry-run commands, HIL testing, and staged activation.

## Scientific Foundations

ApexBEMS implements state-of-the-art methods from:

- **Stochastic Model Predictive Control** — scenario-based dispatch optimization over uncertain price paths
- **Linear Programming** — dispatch optimization with explicit operational constraints
- **Bayesian Model Averaging** — ensemble forecast weighting
- **SHAP Explainability** — optional feature-importance reporting when SHAP is available
- **Arrhenius-Inspired Degradation Models** — temperature-aware battery aging cost estimates

Note: temperature is currently used in degradation-cost calculations. Hard thermal cutoffs and thermal derating constraints are planned integration/model extensions, not implemented behavior in the current monolith.
